iOS Block

Block分三类
NSGlobalBlock －－－ 一个没有对周围变量没有引用的Block
NSMallocBlock
NSStackBlock

对于Global的Block，我们无需多处理，不需retain和copy，因为即使你这样做了，似乎也不会有什么两样。对于Stack的Block，如果不做任何操作，就会向上面所说，随栈帧自生自灭。而如果想让它获得比stack frame更久，那就调用Block_copy()，让它搬家到堆内存上。而对于已经在堆上的block，也不要指望通过copy进行“真正的copy”，因为其引用到的变量仍然会是同一份，在这个意义上看，这里的copy和retain的作用已经非常类似。

copyBlock ＝ ^{ var0,var1,_var2};
weakBlock ＝ ^{ var0,var1,_var2};
blockRight默认创建在栈上，离开方法作用域后，block本身会被抛弃。所以copyBlock会被复制到堆上（NSMallocBlock），weakBlock仍然在栈上（NSStackBlokc）
NSMallocBlock －－ 堆上，离开方法作用域后，仍然存在。
NSStackBlock －－ 栈上，离开方法作用域后，不存在。引用会报错。

临时变量：var0 – copy，备份，互不相关
带blokc前缀的临时变量：__block var1 – 引用，相互关联
属性变量：var2 – 引用，相互关联。不用带上block

_returnBlock = [self createWeakBlock];//离开方法作用域后，_returnBlock会被释放，_returnBlock == nil

TestBlock1 copyCopyBlock = [_copyBlock copy]; //copyCopyBlock 与 _copyBlock是同一份，指向同一个地址

Transitioning to ARC Release Notes－翻译

ARC Rules
1.不能显示调用dealloc，retain，release，retainCount，autorelease。延伸的，不能调用@selector(retain),@selector(release)等等。
因为你不能去主动的释放实例。
在ARC中dealloc方法不需要调用［super dealloc］,这个调用是compiler自动调用的。
可以继续使用CFRetain，CFRelease和其他相关的Core Foundation－style 对象。

2.不能使用NSAllocateObject或者NSDeallocateObject
你应该使用alloc来创造对象，而deallocate由runtime来处理。

3.不能在C structures里使用对象指针
尽量使用Objective-C对象

4.不支持id和void之间的自动转换
当需要id和void之间转换时（Objective-C对象和Core Foundation类型），你需要明确的声明。详情见Managing Toll-Free Bridging

5.不能使用NSAutoreleasePool对象
用@autoreleasepool block代替，比NSAutoreleasePool效率更高

6.不能使用NSZone
NSZone已经被新的Objective－C runtime自动忽略

7.不能用new开头来声明属性
http://blog.csdn.net/meegomeego/article/details/17632515
若是用new开头，那么编译器认为gettre方法是生成一个新的对象
// Won’t work:
@property NSString *newTitle;

// Works:
@property (getter=theNewTitle) NSString *newTitle;

Property Attributes
strong 相当于 retain
weak 相当于assign，当实例被释放后，会自动赋nil（而不再是野指针）
在ARC下，默认strong

Variable Qualifiers
strong 默认，保证对象是alive的 weak 引用，并不保证对象alive，当没有strong的引用时，会自动设nil
unsafe_unretained 同weak，不同的是，当没有strong引用时，不会设nil，而是悬空指针
__autoreleasing 用在函数传参，且这个参数在函数返回之后会释放时。http://blog.txx.im/blog/2013/12/04/about-autoreleasing/

正确的写法
MyClass * __weak myWeakReference;

使用weak的时候需要注意。
1.这种情况下，声明之后就立即被释放了
NSString * weak string = [[NSString alloc] initWithFormat:@”First name: %@”,[self firstName]];
NSLog(@”string: %@”,string);
打印结果 string: (null)

2.当你将一个strong变量做为函数autoreleasing参数传递时，编译器会自动生成一个临时变量。解决方法是，将变量声明为autoreleasing；或将函数参数定义为id strong
-(BOOL)performOperationWithError:(NSError * autoreleasing )error;
/////////////////////
NSError error;
BOOL OK = [myObject performOperationWithError:&error];
if (!OK) {
// Report the error.
// …
/////////////////////

转译为

//////////////////////
NSError __strong error;
NSError __autoreleasing tmp = error;
BOOL OK = [myObject performOperationWithError:&tmp];
error = tmp;
if (!OK) {
// Report the error.
// …
//////////////////////

Use Lifetime Qualifiers to Avoid Strong Reference Cycles
在MRC中 block id x; 不持有x
而在ARC中，block id x; 默认持有x。 若需要有MRC的效果需这样声明unsafe_unretained block id x;更好的方法是使用__weak或置nil来打破循环引用。

ARC Uses a New Statement to Manage Autorelease Pools
在ARC中，应当这样写
@autoreleasepool {

// Code, such as a loop that creates a large number of temporary objects.

}
在入口处建立一个自动释放池，并入栈。
在出口处（break,return,goto,fall-through,等等）自动释放池出栈
当出现异常退出时，自动释放池不会出栈。
@autoreleasepool 的效率比NSAutoreleasePool高6倍

Patterns for Managing Outlets Become Consistent Across Platforms
outlets should be weak, except for those from File’s Owner to top-level objects in a nib file (or a storyboard scene) which should be strong.

Stack Variables Are Initialized with nil
在ARC中，strong，weak，autoreleasing 堆变量会被自动初始化为nil

Use Compiler Flags to Enable and Disable ARC
-fobjc-arc
-fno-objc-arc

Managing Toll-Free Bridging
用的很少，忽略

Oracle RAC

RAC五大特点

  1. 双机并行
    双机热备环境中，始终有一台机器作为备用机，只有当主节点出问题的时候才会切换到备用机上；如果主机一直没有问题，备用机始终处于空闲状态。同时，主机切到备机需要消耗时间。
    RAC环境中，两个节点的集群节点间是一种并行运行的关系，充分利用服务器资源。机器间切换快速，针对存在的绘画之需要数十秒的时间就可以完成失败切换过程，对新回话的创建没有影响。
  2. 高可用性
    高可用性包括两个部分：确保数据不丢失；确保不停机。
    RAC中包含了非常多的高可用特性，主要包含如下几点：
    　　·实现节点间的负载均衡。
    　　·实现失败切换的功能。
    　　·通过Service组件来控制客户端的访问路径。
    　　·集群软件能够自动化管理各个资源，并且有定时的节点状态检测机制，能自动对一些失败的进程以及心跳检测失败的节点进行重启，使其重新恢复到正常的运行状态。
  3. 易伸缩性
    当系统需要更多的处理能力或者需要增加存储时，通过添加另一台服务器或存储设备到集群中，能够在不停机的情况下获得水平的扩展。
    在一个集群中， Clusterware和RAC支持多达100个集群节点。
    11gR2版本中推出了网格即插即用(Grid Plug and Play，GPnP)，可以实现节点的快速添加。
  4. 低成本
    通过多台普通的PC服务器组成一个集群，可以提高集群的处理能力，这样要比采用一台高性能的服务器的成本低很多。
  5. 高吞吐量
    在RAC的架构中，多个实例分布在多个服务器上，能同时打开同一个数据库，而每个实例能够接收相等数量的客户端请求，这样，随着服务器的增加，吞吐量也在不断地增加。

RAC存在的问题：

  1. 稳定性
  2. 高性能
    在目前普遍使用千兆网络的硬件环境中，很多时候系统的数据库从原来的单机迁移至RAC环境，系统的性能反而下降。在这种情况下，数据库管理员应该根据RAC的特点对系统调整提出合理的建议，经过合理的设计、开发，使用RAC是能够提高系统的处理性能的。
